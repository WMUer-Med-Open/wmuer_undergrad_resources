# Content Standard

本仓库的资料要尽量做到可读、可查、可维护。不是每份资料都必须很长，但每份资料都应该让后来者知道它适合谁、怎么用、有什么限制。

## 基本要求

每份 Markdown 资料建议包含：

- 标题。
- 适用年级。
- 适用课程、系统、主题或科室。
- 作者或整理者。
- 更新时间。
- 来源和授权状态。
- 适用场景和局限。

PDF/Word 文档必须配套同名 Markdown 说明，并在说明中补齐这些信息。

## 总结类资料

适用于 `summaries/`。

优秀的医学总结通常具备：

- 有明确主题，不把多个无关课程混在一个文件里。
- 区分“知识点”“考试重点”“个人理解”和“仍需查证的内容”。
- 尽量把基础机制、病理生理、临床表现、诊断思路、治疗原则联系起来。
- 对临床相关内容标注更新时间，必要时提醒读者核对最新指南和课程要求。
- 避免直接复制教材、课件或题库。

不建议收录：

- 只有几行零散口诀且没有上下文的内容。
- 纯粹搬运教材或课件的长段落。
- 没有来源、没有适用范围、无法判断时效性的临床结论。

## 经验贴

适用于 `experience-posts/`。

经验贴可以写得主观，但必须写清楚：

- 背景：年级、课程、考试、科室或时间范围。
- 适用对象：哪些同学可能受益。
- 局限：经验发生的时间、课程制度或个人条件。
- 建议：可以复用的方法，而不是只写情绪或结论。

不建议写成：

- “唯一正确学习路线”。
- 对教师、同学、科室的私人评价。
- 对具体患者或真实病例的可识别描述。

## 文档说明

适用于 `documents/`。

每个 PDF/Word 文档的 Markdown 说明应回答：

- 文件是什么。
- 文件从哪里来。
- 是否原创，或授权状态是什么。
- 适合哪个年级、课程、科室或学习阶段。
- 应该如何使用。
- 有哪些限制和更新时间。

## 语言和格式

- 中文内容为主，标题和文件名可以使用英文。
- 文件名使用小写英文、数字和连字符。
- 表格适合做索引，正文适合解释判断和经验。
- 避免过度排版，优先保证 GitHub 上可读。
